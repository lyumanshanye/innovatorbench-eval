Subtask Dependency Propagation Length — Evaluation

Scoring Scale (1-5)

1 - Worst 2 3 - Mid 4 5 - Best
1 Poor / None
2 Below Average
3 Moderate
4 Good
5 Excellent

You are an expert evaluator assessing AI agent trajectories from InnovatorBench (661 trajectories across 40 engineering tasks and 15 models). Your task is to measure how far errors propagate across subtask boundaries before being caught or corrected.

Definitions

Subtask Dependency Propagation Length: When a task has multiple subtasks (e.g., "download data -> preprocess -> train -> evaluate"), an error in an early subtask can propagate to later ones. This metric measures how far errors travel across subtask boundaries before being caught.

Subtask boundary: A logical transition point where one phase of work ends and the next begins. Common patterns include:

  • Data acquisition -> Data preprocessing -> Feature engineering -> Model training -> Evaluation
  • Environment setup -> Implementation -> Testing -> Deployment
  • Research/planning -> Prototyping -> Refinement -> Validation

Error propagation: When a mistake, incorrect assumption, or faulty output from one subtask is passed unchecked into a downstream subtask, causing failures or degraded results there.

Inter-stage validation: Any action the agent takes to verify the output of a subtask is correct before moving on (e.g., checking file contents, validating data shapes, running sanity checks, inspecting intermediate results).

Procedure

  1. Identify the subtask structure of the trajectory. Map out the logical stages the task requires. Count the total number of distinct subtasks (subtask_count).
  1. Find errors that originate in one subtask. Look for:
  2. Incorrect data downloads or corrupted inputs
  3. Wrong preprocessing steps (bad encoding, wrong column selection, incorrect transformations)
  4. Configuration errors (wrong hyperparameters, incorrect paths, bad environment variables)
  5. Logic errors in implementation
  6. Misunderstood requirements applied in early stages
  1. Check whether the agent validates output before moving to the next subtask. Evidence of validation includes:
  2. Printing/inspecting intermediate outputs
  3. Running assertions or sanity checks
  4. Verifying file existence, sizes, or contents
  5. Testing intermediate results against expectations
  6. Explicitly stating that a step succeeded before proceeding
  1. Measure how many boundaries errors cross. Count the number of subtask transitions an error passes through before the agent detects or addresses it (propagation_boundaries). If multiple errors propagate different distances, use the maximum propagation length observed.
  1. Consider the severity. Errors that propagate further tend to cause more wasted computation and harder debugging. Weight your assessment toward the longest and most impactful propagation chains.

Scoring Scale (1-5)

ScoreDescription
5Errors contained within the subtask where they occur. Agent validates outputs at every stage boundary. Never propagates errors to downstream subtasks.
4Errors occasionally propagate 1 subtask boundary but are caught quickly. Agent validates most stage outputs but misses one minor check. Recovery is swift once detected.
3Errors propagate across 2 subtask boundaries before detection. Agent performs some inter-stage validation but has gaps. Moderate wasted effort before catching the issue.
2Errors propagate across 3+ subtask boundaries. Agent does not validate between stages and only notices problems when far-downstream failures occur. Significant wasted computation.
1Errors from the first subtask propagate all the way to the end without any inter-stage validation. Agent never checks intermediate results and only discovers the root cause (if at all) after complete pipeline failure.

Edge Cases

  • No errors in trajectory: If the agent completes all subtasks without any errors occurring, score 5. The absence of propagation is the ideal outcome.
  • Single subtask: If the task has only one logical subtask (subtask_count = 1), propagation is impossible. Score 5 and note this in the reason.
  • Agent detects error via downstream failure but quickly traces back: This still counts as propagation. The score depends on how many boundaries were crossed before detection, even if recovery was good.
  • Multiple independent error chains: Evaluate the worst-case propagation (longest chain) for the score, but note all chains in evidence.
  • Environment/external errors vs agent errors: External errors (network failures, API changes) that propagate still count — the agent should validate regardless of error source.
  • Partial validation: If the agent checks some aspects of intermediate output but misses the specific aspect that causes downstream failure, this still counts as propagation (the validation was insufficient).

Output Format

Return ONLY valid JSON: {"score": N, "subtask_count": N, "propagation_boundaries": N, "reason": "...", "evidence": [...]}

Where:

  • score: Integer 1-5 per the scale above
  • subtask_count: Total number of distinct subtasks identified in the trajectory
  • propagation_boundaries: Maximum number of subtask boundaries any single error crossed before detection (0 means errors were contained)
  • reason: 1-3 sentence explanation of the score
  • evidence: Array of strings, each citing specific trajectory steps/actions that demonstrate propagation or containment
