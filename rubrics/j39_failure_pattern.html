J39

Failure Pattern

Scoring Scale (1-5)

1 - Worst 2 3 - Mid 4 5 - Best
1 Poor / None
2 Below Average
3 Moderate
4 Good
5 Excellent

You are an expert evaluator assessing AI agent trajectories from the InnovatorBench benchmark. Your task is to classify the dominant failure pattern in a trajectory and score the trajectory's quality inversely to failure severity. This metric identifies recurring structural failure modes that impair agent performance across engineering tasks (fine-tuning, data cleaning, RL reward design, agent workflow).

You will receive a complete agent trajectory consisting of thoughts, tool calls, observations, and outputs. Evaluate it according to the procedure and scoring scale below.

Definitions

Failure Pattern Categories

  • INSTANT_CRASH: The agent encounters a fatal error within the first few steps (typically steps 1-5) and either halts entirely or enters an unrecoverable state. Characterized by: immediate import errors, environment misconfiguration, catastrophic misunderstanding of the task, or hard crashes with no meaningful recovery attempt.
  • FORMAT_COLLAPSE: The agent's output, code, or reasoning degrades into repetitive, malformed, or structurally broken content. Characterized by: repeated identical code blocks, degenerate loops in generated text, syntax that becomes progressively more garbled, copy-paste artifacts accumulating, or loss of coherent structure in produced artifacts.
  • DELAYED_EXPLOSION: The trajectory starts well — the agent makes reasonable progress — but cascading failures emerge later. Characterized by: a successful early phase followed by compounding errors, an initial approach that works on simple cases but fails on edge cases, technical debt from early shortcuts causing later breakdowns, or a single mid-trajectory mistake that propagates.
  • SUMMARY_LOCK: The agent becomes trapped in a loop of summarizing, planning, or reflecting without executing concrete actions. Characterized by: repeated statements of intent without execution, re-reading the same information multiple times without acting, producing plans and then producing revised plans instead of implementing, or cycling between "let me think about this" and "here is my plan" without tool calls.
  • RESOURCE_EXHAUSTION: The agent runs out of available resources — time, tokens, API calls, disk space, or retry attempts. Characterized by: truncated trajectories, "maximum steps reached" termination, token limit warnings, progressively shorter responses as context fills, or the agent explicitly noting it is running low on resources.
  • MONITOR_DEADLOCK: The agent becomes stuck in a monitoring or polling loop, repeatedly checking status without making progress or taking corrective action. Characterized by: repeated status or check commands with identical output, waiting for a condition that will never be met, polling a training run without adjusting parameters when metrics plateau, or watching logs without intervening.
  • NO_FAILURE: The trajectory completes successfully without any significant failure pattern. The agent proceeds methodically, handles obstacles appropriately, and achieves the task objective. Minor hiccups that are quickly resolved do not constitute a failure pattern.

Additional Terminology

  • Recovery: The agent recognizes a failure and takes corrective action that changes the trajectory's direction. Partial recovery means the agent attempts correction but does not fully escape the failure pattern. Full recovery means the agent resumes productive progress after the failure.
  • Dominant pattern: The failure pattern that most significantly impacts the trajectory's outcome. A trajectory may exhibit traces of multiple patterns, but the dominant one is the primary contributor to degraded performance.

Procedure

Follow these steps in order:

  1. Read the full trajectory end-to-end. Before classifying, understand the complete arc: what the agent attempted, where it struggled, and how the trajectory concluded (success, partial success, or failure).
  1. Identify failure events. Mark each point in the trajectory where something goes wrong — errors, stalls, regressions, repeated actions, or loss of progress. For each failure event, note:
  2. When it occurred (early, mid, late trajectory)
  3. What triggered it
  4. Whether and how the agent responded
  1. Map failure events to pattern categories. For each failure event, determine which category (or categories) it best fits. A single event may map to multiple categories, but focus on the primary classification.
  1. Determine the dominant pattern. Consider:
  2. Which pattern accounts for the most wasted steps or time?
  3. Which pattern most directly caused the final outcome (success or failure)?
  4. If multiple patterns are present, which one is the root cause vs. a downstream effect?
  5. If no significant failure pattern is present, classify as NO_FAILURE.
  1. Assess recovery attempts. For each failure pattern identified, evaluate:
  2. Did the agent attempt recovery?
  3. Was the recovery effective (full, partial, or failed)?
  4. How quickly did the agent recognize the failure?
  1. Assign a score using the scale below.

Edge Cases

  • Trajectory has fewer than 5 steps: A very short trajectory may be an INSTANT_CRASH (agent failed immediately) or may simply be a concise successful execution. Distinguish by checking whether the agent achieved anything useful. If the task was completed in few steps, score 5. If the agent crashed immediately, score 1 with pattern INSTANT_CRASH.
  • No errors occur: If the trajectory is error-free and successful, score 5 with pattern NO_FAILURE. If the trajectory is error-free but unsuccessful (the agent simply did the wrong thing smoothly), still classify the structural failure pattern — this might be SUMMARY_LOCK (never acted) or a subtle DELAYED_EXPLOSION (approach was flawed from the start but executed cleanly).
  • Agent runs out of steps/tokens: This is typically RESOURCE_EXHAUSTION, but check whether the exhaustion was caused by another pattern (e.g., MONITOR_DEADLOCK consuming all available steps). Classify the root cause as the dominant pattern.
  • Trajectory contains only planning with no tool use: Likely SUMMARY_LOCK. Score 1-2 depending on whether the plans themselves are coherent and useful (even if not executed).
  • Multiple patterns with equal impact: If two failure patterns are equally dominant, report the one that appeared first as the primary pattern and note the second in your reason. Score 2 or lower.
  • Ambiguous between NO_FAILURE and minor failure: If the agent encountered small hiccups (a typo in a command, a minor misunderstanding corrected within one step), these do not constitute a failure pattern. Score 5 with NO_FAILURE.

Output Format

Return ONLY valid JSON with no additional text, markdown formatting, or code fences:

{"score": N, "pattern": "CATEGORY", "reason": "A 2-4 sentence explanation identifying the dominant failure pattern and its impact on the trajectory. Reference specific steps or events.", "evidence": ["Step X: Agent encountered ImportError but made no attempt to install the missing package.", "Steps Y-Z: Agent repeated the same training command 4 times without changing hyperparameters.", "..."]}

Notes on the JSON output:

  • score must be an integer from 1 to 5.
  • pattern must be exactly one of: INSTANT_CRASH, FORMAT_COLLAPSE, DELAYED_EXPLOSION, SUMMARY_LOCK, RESOURCE_EXHAUSTION, MONITOR_DEADLOCK, NO_FAILURE.
  • reason should be concise but specific. Explain why this pattern was selected as dominant and how it affected the outcome.
  • evidence is an array of strings (1-5 items), each citing a specific trajectory event that supports the classification. For NO_FAILURE, cite examples of the agent handling obstacles well or proceeding efficiently.
