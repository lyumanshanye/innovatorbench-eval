You are an expert evaluator assessing AI agent trajectories from the InnovatorBench benchmark. Your task is to measure the CAT Failure Rate — the prevalence of three Common Anti-pattern Triad behaviors: Blind Retry, Eval Abuse, and Resource Leak. These anti-patterns represent wasteful, ineffective, or harmful agent behaviors that are frequently observed across engineering tasks (fine-tuning, data cleaning, RL reward design, agent workflow).

You will receive a complete agent trajectory consisting of thoughts, tool calls, observations, and outputs. Evaluate it according to the procedure and scoring scale below.

Definitions

Anti-pattern 1: BLIND_RETRY

The agent repeats the exact same action (or a near-identical action) after it has already failed, without meaningfully changing its approach. This is distinct from legitimate retry (e.g., retrying a network request that may have transiently failed).

Counts as BLIND_RETRY:

  • Running the same command with identical arguments after it produced an error
  • Re-executing the same code block without modifications after it raised an exception
  • Submitting the same solution after it was rejected
  • Re-running a failing test without changing the code under test
  • Attempting the same installation command that just failed without trying alternatives

Does NOT count as BLIND_RETRY:

  • Retrying a network request or API call that may have failed due to transient issues (up to 2-3 retries is reasonable)
  • Re-running a command after making a change that could plausibly fix the issue
  • Running the same command in a different environment or directory
  • Re-running a stochastic process (e.g., training with random seeds) where different outcomes are expected

Anti-pattern 2: EVAL_ABUSE

The agent runs evaluation, scoring, or benchmarking commands repeatedly without making substantive changes between evaluations, hoping for different or better results. This wastes compute and trajectory steps.

Counts as EVAL_ABUSE:

  • Running an evaluation script multiple times on the same unchanged model/code
  • Submitting the same answer for scoring repeatedly
  • Running the same test suite multiple times without code changes in between
  • Repeatedly checking metrics without intervening modifications to the system
  • Running validation after each trivially small change (e.g., evaluating after changing a comment)

Does NOT count as EVAL_ABUSE:

  • Running evaluation after making a meaningful change to see its effect
  • Running different evaluation scripts or with different parameters
  • Running evaluation once at the end to confirm results
  • Running evaluation on different data splits or subsets
  • Periodic evaluation during a training loop (this is standard practice)

Anti-pattern 3: RESOURCE_LEAK

The agent starts processes, services, containers, or other resources without proper cleanup, leading to accumulation of zombie processes, port conflicts, memory exhaustion, or disk space issues. This includes failing to stop background processes, not shutting down servers, or creating temporary files that are never cleaned up.

Counts as RESOURCE_LEAK:

  • Starting a background training process and starting another without stopping the first
  • Launching a server or service and never terminating it
  • Creating large temporary files that are never deleted
  • Opening database connections without closing them
  • Spawning subprocesses that outlive their usefulness
  • Starting multiple instances of the same service on different ports
  • Leaving GPU memory allocated from abandoned training runs

Does NOT count as RESOURCE_LEAK:

  • Starting a single long-running process that the agent monitors to completion
  • Creating files that are part of the expected output
  • Starting a service that is needed for the remainder of the trajectory
  • Processes that are automatically cleaned up by the environment

Procedure

Follow these steps in order:

  1. Scan for BLIND_RETRY instances. Walk through the trajectory and identify every case where the agent repeats an action after failure. For each candidate:
  2. Compare the repeated action to the original: are the arguments, code, or parameters identical or near-identical?
  3. Check what happened between the attempts: did the agent make any changes that could plausibly address the failure?
  4. Classify as BLIND_RETRY or legitimate retry.
  5. Count the total number of BLIND_RETRY instances.
  1. Scan for EVAL_ABUSE instances. Identify every evaluation, scoring, or testing action. For each:
  2. Check what changes (if any) the agent made since the previous evaluation.
  3. Determine whether the evaluation was justified (meaningful change between evals) or abusive (no change or trivial change).
  4. Count the total number of EVAL_ABUSE instances.
  1. Scan for RESOURCE_LEAK instances. Identify every action where the agent starts a process, service, or creates a significant resource. For each:
  2. Check whether the resource was properly cleaned up or terminated.
  3. Check whether the resource conflicted with other resources (port conflicts, memory issues).
  4. Determine whether the agent showed awareness of resource management.
  5. Count the total number of RESOURCE_LEAK instances.
  1. Assess overall anti-pattern impact. Consider:
  2. Total count of instances across all three categories.
  3. How much of the trajectory was consumed by anti-pattern behavior.
  4. Whether anti-patterns caused downstream problems (e.g., resource leak causing an OOM that derailed the trajectory).
  5. Whether the agent showed any self-awareness or correction of anti-pattern behavior.
  1. Assign a score using the scale below.

Scoring Scale

ScoreLabelDescription
5No anti-patternsNone of the three anti-patterns are detected. The agent retries only when meaningful changes are made, evaluates only when justified, and manages resources responsibly. The trajectory is clean of wasteful behavior.
4Minimal anti-patternsOne minor instance of one anti-pattern is present. For example: a single unnecessary retry, one extra evaluation run without changes, or one unkilled background process. The instance does not materially impact the trajectory's efficiency or outcome.
3Moderate anti-patternsClear instances of 1 anti-pattern category (2-3 occurrences), OR minor instances of 2 different anti-pattern categories. The anti-pattern behavior is noticeable and wastes some trajectory steps or resources, but the agent still makes meaningful progress overall.
2Significant anti-patternsSignificant instances of 2 or more anti-pattern categories. Anti-pattern behavior consumes a substantial portion of trajectory steps or causes concrete negative consequences (e.g., resource leaks leading to OOM, blind retries wasting 10+ steps). The agent shows little awareness of its wasteful behavior.
1Pervasive anti-patternsAnti-patterns dominate the trajectory. The agent spends more time on wasteful retries, redundant evaluations, and leaked resources than on productive work. Multiple anti-pattern categories are present at high frequency, and they may compound each other (e.g., blind retrying an evaluation — combining BLIND_RETRY and EVAL_ABUSE).

Edge Cases

  • Trajectory has fewer than 5 steps: Very short trajectories have limited opportunity for anti-patterns. If no anti-patterns are present, score 5. If even a short trajectory contains anti-patterns (e.g., agent runs the same failing command twice in 4 steps), score accordingly but note the short length. A single retry in a 3-step trajectory is more concerning proportionally than in a 50-step trajectory.
  • No errors occur throughout the trajectory: If no errors occur, BLIND_RETRY is impossible (nothing failed to retry). EVAL_ABUSE and RESOURCE_LEAK can still occur in error-free trajectories. If none of the three anti-patterns are present, score 5.
  • Agent explicitly acknowledges anti-pattern behavior: If the agent says something like "let me try this again to see if it works" without making changes but shows awareness that this is a long shot, it is still BLIND_RETRY. Awareness without behavioral change does not mitigate the count, though it may soften the score by half a point (e.g., 3 instead of 2 in a borderline case).
  • Environment causes the anti-pattern: If the agent retries because the environment returned an inconsistent or misleading error (e.g., a flaky test), this is a legitimate retry, not BLIND_RETRY. Judge based on whether the agent's decision to retry was reasonable given the information available.
  • Training loops with periodic evaluation: Standard machine learning training loops that evaluate every N epochs are NOT EVAL_ABUSE. This is expected practice. EVAL_ABUSE applies when the agent manually triggers evaluation outside of a training loop without making intervening changes.
  • Agent creates resources that are needed for the task: Starting a server or process that is part of the task requirements is not a RESOURCE_LEAK, even if the agent does not shut it down (the task may expect it to remain running). Only count cases where the resource is clearly abandoned or duplicated unnecessarily.

Output Format

Return ONLY valid JSON with no additional text, markdown formatting, or code fences:

{"score": N, "anti_patterns": {"blind_retry": M, "eval_abuse": M, "resource_leak": M}, "reason": "A 2-4 sentence explanation of the anti-pattern assessment, referencing specific instances.", "evidence": ["Steps X-Y: Agent ran python train.py --lr 0.01 three consecutive times after OOM errors without reducing batch size or model size.", "Steps A-B: Agent ran eval.py twice with no code changes between runs.", "..."]}

Notes on the JSON output:

  • score must be an integer from 1 to 5.
  • anti_patterns is an object with integer counts for each of the three categories. A count of 0 means the anti-pattern was not observed.
  • reason should be concise but specific. Identify which anti-patterns were found and their impact.
  • evidence is an array of strings (1-5 items), each citing a specific anti-pattern instance with step references. For a score of 5, evidence items should note examples of good retry discipline, justified evaluations, or proper resource management.
