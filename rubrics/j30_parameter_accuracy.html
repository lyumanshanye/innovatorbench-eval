Parameter Value Accuracy — Evaluation

Scoring Scale (1-5)

1 - Worst 2 3 - Mid 4 5 - Best
1 Poor / None
2 Below Average
3 Moderate
4 Good
5 Excellent

You are an expert evaluator assessing AI agent trajectories from InnovatorBench (661 trajectories across 40 engineering tasks and 15 models). Your task is to evaluate how accurately the agent supplies parameter values when calling tools, functions, or commands throughout its trajectory.

Definitions

Parameter Value Accuracy measures whether the argument values the agent passes to tools and commands are correct, appropriate, and functional. This includes:

  • File paths: Do referenced paths actually exist or correspond to real locations in the project?
  • Function/method names: Are they spelled correctly and refer to actual callable entities?
  • Flags and options: Are CLI flags valid for the command being invoked (e.g., --recursive vs --recurse)?
  • Numeric parameters: Are values reasonable for the context (e.g., timeout values, batch sizes, port numbers)?
  • Format strings: Do patterns match the expected format (e.g., date formats, regex patterns, template syntax)?
  • Enum/choice values: Are string arguments from the valid set of accepted values?
  • Type correctness: Are integers passed where integers are expected, arrays where arrays are expected, etc.?

This metric does NOT evaluate whether the agent chose the right tool/command in the first place (that is a separate concern). It only evaluates whether, given the tool/command the agent chose to invoke, the parameters supplied are correct.

Procedure

  1. Enumerate all tool/command calls in the trajectory. For each call, list the parameters provided.
  2. For each parameter, assess correctness:
  3. Does the value refer to something that exists (paths, names, identifiers)?
  4. Is the value syntactically valid for the parameter type?
  5. Is the value semantically appropriate for the operation?
  6. Would the value cause the tool call to fail or produce unintended results?
  7. Classify parameter errors by severity:
  8. Critical: Causes the tool call to fail entirely (wrong path, nonexistent function, invalid flag)
  9. Moderate: Tool call succeeds but produces wrong/suboptimal results (wrong branch name, incorrect regex)
  10. Minor: Technically imprecise but functionally equivalent (extra whitespace, verbose path vs relative path)
  11. Consider self-correction: If the agent immediately recognizes and corrects a parameter error in the next step, reduce the severity weight of that error.
  12. Assign a score based on the overall pattern of parameter accuracy across the full trajectory.

Scoring Scale (1-5)

ScoreDescription
5All tool calls have correct, appropriate parameters. Every path exists, every function name is spelled correctly, every flag is valid, every numeric value is reasonable. Zero or near-zero parameter issues across the entire trajectory.
41-2 minor parameter issues throughout the trajectory (e.g., slightly suboptimal but still functional values, a path that works but isn't the canonical form). No parameter errors cause tool call failures.
3Several parameter errors present, some causing tool call failures that require correction. The agent recovers from these failures but wastes steps due to incorrect parameters. Roughly 10-20% of tool calls have parameter issues.
2Frequent parameter errors causing cascading issues. Wrong paths lead to wrong file reads, which lead to wrong edits. The agent struggles to supply correct parameters and spends significant trajectory time on parameter-related failures.
1Pervasive wrong parameters throughout the trajectory. Wrong file paths, misspelled function names, invalid flags, unreasonable numeric values. The majority of tool calls have at least one incorrect parameter, severely impeding progress.

Edge Cases

  • Exploratory calls: If the agent intentionally tries a path to check if it exists (e.g., ls /possible/path), this is not a parameter error even if the path doesn't exist — it is information gathering.
  • Ambiguous correctness: If the task description is ambiguous about the correct parameter value, give the agent benefit of the doubt if the chosen value is reasonable.
  • Environment differences: If the agent uses parameters correct for one environment but wrong for the actual execution environment (and had no way to know), reduce severity.
  • Template/placeholder values: If the agent uses placeholder values (e.g., <YOUR_API_KEY>) where real values are needed, count as parameter errors unless the task explicitly calls for templating.
  • Corrected errors: An error that is immediately detected and corrected in the very next step counts at half weight compared to an error that persists or cascades.

Output Format

Return ONLY valid JSON: {"score": <integer 1-5>, "reason": "<2-4 sentence justification referencing specific parameter errors or lack thereof>", "evidence": ["<quote or description of specific tool call with parameter issue>", "<quote showing correct parameter usage or correction>", "..."]}
