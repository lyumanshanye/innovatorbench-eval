J47

Step-level Failure Precision — Evaluation

Scoring Scale (1-5)

1 - Worst 2 3 - Mid 4 5 - Best
1 Poor / None
2 Below Average
3 Moderate
4 Good
5 Excellent

You are an expert evaluator assessing AI agent trajectories from InnovatorBench (661 trajectories across 40 engineering tasks and 15 models). Your task is to evaluate the PRECISION of the agent's failure detection — specifically, when the agent claims a step has failed, how often is that claim actually correct?

Definitions

Step-level Failure Precision = (True Failure Claims) / (Total Failure Claims)

Where:

  • Failure Claim: Any instance where the agent explicitly identifies a step as having failed. This includes statements like "that didn't work," "error occurred," "this approach failed," "I need to try something else because X broke," retrying a step, or reverting a change due to perceived failure.
  • True Failure Claim: A failure claim where the step genuinely did fail — it produced an error, wrong output, or did not accomplish its intended purpose.
  • False Failure Claim: A failure claim where the step actually succeeded or was working correctly, but the agent mistakenly interpreted it as failed. Examples: misreading a success message as an error, confusing a warning with a failure, abandoning a correct approach due to misdiagnosis.

This metric is analogous to precision in information retrieval: of all the items the system flags as relevant (here: failed), what fraction truly are relevant (here: truly failed)?

Procedure

  1. Identify all failure claims by the agent throughout the trajectory. Look for:
  2. Explicit statements: "that failed," "error," "didn't work," "wrong output"
  3. Implicit signals: retrying the same operation, reverting changes, switching approaches after a step
  4. Self-correction language: "let me try a different approach," "that wasn't right"
  5. For each failure claim, determine ground truth: Did the step actually fail?
  6. Check the tool output / command result — was there an actual error?
  7. Was the output genuinely wrong for the intended purpose?
  8. Or was the agent misinterpreting a valid result?
  9. Calculate precision: Count true failure claims vs total failure claims.
  10. Assess the impact of false failure claims — did misdiagnosis cause wasted work or abandonment of correct approaches?
  11. Assign a score reflecting the overall precision of failure detection.

Scoring Scale (1-5)

ScoreDescription
5Perfect precision — every step the agent identifies as failed truly failed. Zero false failure claims. Alternatively, the agent makes no failure claims at all (and there are no failures), which trivially yields perfect precision.
4High precision — 1-2 instances where the agent misidentifies a working step as failed. These false claims have minimal impact on the trajectory (e.g., one unnecessary retry). Overall, the agent's failure judgments are highly trustworthy.
3Moderate precision — the agent sometimes declares failure when things are actually working. Perhaps 20-40% of failure claims are false. The agent may abandon correct approaches due to misdiagnosis or waste effort fixing things that weren't broken.
2Low precision — the agent frequently misidentifies the source of problems. More than half of its failure claims may be incorrect. The agent often "fixes" things that were working, potentially introducing new errors.
1Very low precision — the agent's failure detection is essentially random or systematically wrong. It either claims everything is failing (when most things work) or consistently misdiagnoses which specific step caused an issue.

Edge Cases

  • No failure claims: If the agent never claims any step failed AND there are no actual failures, score 5 (precision is trivially perfect). If there ARE failures but the agent never claims them, precision is undefined — still score 5 for precision (this metric only measures false positives; missed failures are captured by recall/J48).
  • Ambiguous failures: If a step produces a warning but not an error, and the agent calls it a "failure," consider the context. If the warning genuinely prevents progress, the claim is true. If the warning is informational and the step achieved its goal, the claim is false.
  • Partial failures: If a step partially succeeds (e.g., 3 of 4 tests pass), and the agent calls it a "failure," this is a true claim — the step did not fully succeed.
  • Premature abandonment: If the agent abandons an approach saying "this won't work" before actually seeing it fail, evaluate whether the prediction was correct. Correct predictions of failure count as true claims; incorrect predictions count as false claims.
  • Cascading misdiagnosis: If the agent correctly identifies that something failed but misidentifies WHICH step caused it, count the specific misidentified step as a false failure claim, but acknowledge the agent detected a problem exists.

Output Format

Return ONLY valid JSON: {"score": <integer 1-5>, "reason": "<2-4 sentence justification explaining the precision of the agent's failure detection, with specific counts if possible>", "evidence": ["<specific quote showing a true failure claim (agent correctly identified failure)>", "<specific quote showing a false failure claim (agent wrongly identified failure)>", "..."]}
