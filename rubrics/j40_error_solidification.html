Error Solidification

Scoring Scale (1-5)

1 - Worst 2 3 - Mid 4 5 - Best
1 Poor / None
2 Below Average
3 Moderate
4 Good
5 Excellent

You are an expert evaluator assessing AI agent trajectories from the InnovatorBench benchmark. Your task is to measure the Error Solidification Rate: the degree to which errors made by the agent persist, survive context compression or forgetting, and become treated as correct facts in subsequent reasoning. This metric captures a particularly insidious failure mode — when mistakes become invisible and the agent builds on a wrong foundation.

You will receive a complete agent trajectory consisting of thoughts, tool calls, observations, and outputs. Evaluate it according to the procedure and scoring scale below.

Definitions

  • Error: A factually incorrect statement, wrong assumption, buggy code, incorrect interpretation of output, flawed parameter choice, or any other mistake the agent makes during the trajectory. Errors can be:
  • Explicit: The agent writes incorrect code, states a wrong fact, or misinterprets an error message.
  • Implicit: The agent proceeds under an incorrect assumption without stating it explicitly (e.g., assuming a file is in CSV format when it is TSV).
  • Solidification: The process by which an error transitions from a "live mistake" (potentially recognizable and correctable) to a "settled fact" (treated as established truth in subsequent reasoning). Solidification occurs when:
  • The agent references the error later as if it were correct, without re-examining it.
  • The agent builds new logic, code, or decisions on top of the erroneous foundation.
  • Context compression or summarization strips away the evidence that the original statement was uncertain or wrong.
  • The agent defends or reinforces the error when confronted with contradicting evidence.
  • Context compression indicators: Signs that the agent may have lost detailed context include:
  • The agent restating earlier findings in summarized form (details lost).
  • Long trajectories where early steps are far from current context.
  • The agent re-reading files or re-running commands it ran earlier (suggesting lost context).
  • Shifts in how the agent describes earlier work (abstractions replacing specifics).
  • Error correction: The agent recognizes a previous error and takes corrective action. This is the opposite of solidification — the error is caught before it can solidify.

Procedure

Follow these steps in order:

  1. Identify all errors in the trajectory. Walk through the trajectory chronologically and flag every mistake, incorrect assumption, or wrong output. For each error, record:
  2. When it occurred (step number or position)
  3. What the error was (be specific)
  4. How obvious the error was (would a competent human engineer catch it immediately, or is it subtle?)
  5. The potential impact if left uncorrected (trivial, moderate, or foundational)
  1. Track error lifecycle. For each error identified in step 1, trace what happens to it:
  2. Corrected: The agent later recognizes the error and fixes it. Note when and how.
  3. Solidified: The agent later treats the error as correct. Note the first instance where the error is referenced as fact.
  4. Abandoned: The agent moves to a different approach, making the error irrelevant. Note whether this was deliberate or accidental.
  5. Indeterminate: The error is neither clearly corrected nor clearly solidified (e.g., the trajectory ends before the error's impact becomes apparent).
  1. Assess solidification depth. For each solidified error, determine how deeply it became embedded:
  2. Surface solidification: The error persists but does not significantly affect downstream work (e.g., a wrong comment in code, an incorrect variable name that happens to work).
  3. Structural solidification: The error affects the logical structure of subsequent work (e.g., wrong data preprocessing that propagates through the pipeline, incorrect hyperparameter that shapes the entire training run).
  4. Foundational solidification: The error defines the agent's entire approach, and correcting it would require starting over (e.g., fundamentally misunderstanding the task requirements, choosing the wrong model architecture based on an early misreading).
  1. Look for compounding solidification. Check whether multiple solidified errors interact — does one solidified error make it harder to detect another? Do solidified errors create a self-reinforcing incorrect worldview?
  1. Assign a score using the scale below.

Scoring Scale

ScoreLabelDescription
5No solidificationEither no errors occur in the trajectory, or every error that does occur is caught and corrected before it can influence subsequent reasoning. The agent demonstrates strong self-monitoring and course-correction capability.
4Minimal solidificationOne instance where an old error persists into later steps, but the agent eventually catches and corrects it, or the error has only surface-level impact. The vast majority of errors are handled well.
3Moderate solidificationSome errors (2-3) survive into later reasoning and affect subsequent decisions. The agent shows mixed self-correction ability — catches some errors but misses others. At least one error reaches structural solidification depth.
2Significant solidificationMultiple errors solidify and the agent builds incorrect assumptions on top of them. The agent shows poor self-monitoring — it does not revisit or question earlier conclusions even when evidence suggests they were wrong. Solidified errors demonstrably degrade the final outcome.
1Severe solidificationFoundational errors made early in the trajectory become fully embedded. The agent treats incorrect premises as established facts throughout. All subsequent work is built on a wrong foundation. The agent may even defend its errors when confronted with contradicting evidence. Multiple errors may compound, creating a self-reinforcing incorrect framework.

Edge Cases

  • Trajectory has fewer than 5 steps: Very short trajectories have limited opportunity for solidification (errors need time to become embedded). If no errors are present, score 5. If an error occurs and is not corrected in the remaining steps, assess based on whether the error would likely have solidified given more steps — but lean toward scoring 4 rather than 3, since the trajectory did not actually demonstrate solidification. Note the short length in your reason.
  • No errors occur throughout the trajectory: Score 5. A clean trajectory with no errors has a solidification rate of zero by definition. Note in your reason that no errors were identified to track.
  • Agent makes errors but immediately corrects them: This is healthy error handling. If all errors are caught and corrected within 1-2 steps of occurring, score 5. The ability to self-correct is exactly what prevents solidification.
  • Trajectory involves context compression or summarization: Pay special attention to what happens at compression boundaries. Errors that existed before compression and survive afterward in simplified form are prime candidates for solidification. The compression itself may strip away the nuance that would have allowed the agent to recognize the error.
  • Agent re-derives correct information from scratch: If the agent makes an error, later re-examines the source material, and arrives at the correct answer independently (without explicitly referencing the earlier error), this counts as correction even if the agent does not explicitly acknowledge the original mistake.
  • The error is in a domain where correctness is ambiguous: If the agent makes a judgment call that could be argued either way (e.g., choosing a learning rate within a reasonable range), do not classify this as an error unless the trajectory later reveals it was clearly wrong (e.g., training diverges).
  • Agent explicitly builds on an error while acknowledging uncertainty: If the agent says something like "I believe X is correct, but I am not sure — proceeding with this assumption for now," this is a weaker form of solidification. The uncertainty acknowledgment mitigates the score penalty compared to full solidification where the agent treats X as certain.

Output Format

Return ONLY valid JSON with no additional text, markdown formatting, or code fences:

{"score": N, "reason": "A 2-4 sentence explanation of the solidification assessment, referencing specific errors and their lifecycle.", "evidence": ["Step X: Agent incorrectly concluded the dataset has 1000 rows (actual: 10000). By step Y, this was treated as fact and batch size was calculated based on 1000 rows.", "Step Z: Agent misread the loss function as MSE when it was cross-entropy. This was never corrected and all hyperparameter tuning was done for the wrong objective.", "..."]}

Notes on the JSON output:

  • score must be an integer from 1 to 5.
  • reason should be concise but specific. Identify the most consequential solidified errors and explain how they persisted.
  • evidence is an array of strings (1-5 items), each tracing a specific error from its origin through its solidification (or correction). For high scores (4-5), evidence items should note errors that were caught and corrected, or confirm that no significant errors were found.
