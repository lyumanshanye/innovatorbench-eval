01研究动机

当前 Agent 评测的困境

随着 LLM Agent 的快速发展，各类 Agent 系统已能深度整合外部工具、接管通讯与操作界面。然而现有评测方法存在严重局限：

  • 整体指标的遮蔽效应：大多数 benchmark 仅报告端到端任务的整体成功率，无法揭示 Agent 在不同维度上的优劣。InterpretEval (Fu et al., 2020) 显示，一个 NER 模型整体 F1=85 可能在长实体上只有45。Agent 评测同样如此。
  • 侧重于关注最终交付结果：在真实场景中的任务往往是持续数天的复杂过程，Agent 评测也不应只看任务是否完成，而应审视决策链路每一步。
  • 缺少对步骤级交付物的评估：Agent 完成 long-horizon 任务时，在中间阶段需要及时向人求证以完成任务，模型的很多错误是因为过于自信直接完成任务所致。

v4 NEW 工具使用评测的粗粒度问题：TRAJECT-Bench 揭示现有 benchmark 仅统计"是否调用了正确工具"，忽视了参数正确性、调用顺序合理性、冗余调用等关键维度。在 1228 个工具的评测中，Agent 在 Exact Match 仅 23.5%，但 Inclusion（包含正确工具）达 67.2%——说明 Agent "知道用什么"但"不知道怎么用好"。

v4 NEW 失败定位缺乏层级结构：CodeTracer 提出将轨迹组织为 Goal → Sub-goal → Action 的层级树（trace tree），在 step-level 达到 48% F1 的失败定位精度。对比 v3 的扁平因果回溯，层级化结构可以精确区分"哪个子目标失败"而非仅定位"哪一步出错"。

v4 NEW 自动评估的系统性偏差：AlpacaEval-LC 发现，不控制输出长度时 LLM-as-Judge 的排行榜与人类偏好的 Spearman 相关仅 0.94，控制后升至 0.98。在 Agent 评测中，token 消耗量、步骤数等变量同样可能引入类似偏差。

02问题定义

研究目标

本项目的核心目标是构建一套面向真实场景的 Long-Horizon Agent 多维度评测体系，包含三个层次：

  1. 评测环境：构建贴近实际应用场景、支持深度交互与复杂操作的高效 Agent 评测环境。
  2. 评测任务集：设计面向长决策链路 (Long-Horizon) 的真实场景 Agent 评测任务集，覆盖多个职业领域。
  3. 评测指标：建立针对 Agent 决策链路、工具使用、错误纠正及时间效率的多维度评测指标。

关键研究问题

RQ1

如何定义 Agent 的"属性"，可以更全面的揭示 Agent 在真实任务场景下的能力分级？

RQ2

如何对 Agent 多步决策链路做步骤级评估，而非仅看最终完成情况？

RQ3

如何构建贴近生产环境的 Long-Horizon 评测任务，暴露 Agent 在真实场景中的失败模式？

RQ4

评测结果能否反哺 Agent 训练？

v4 NEW v4 新增两个研究问题：

RQ5 NEW

如何消除自动评估器中的系统性偏差（如 token 消耗量、步骤数量对评分的混淆效应），使评测结果更接近真实能力排序？

RQ6 NEW

Agent 在工具使用轨迹的不同粒度（工具选择 / 参数正确性 / 调用顺序 / 整体满意度）上的能力瓶颈分别在哪里？

03方法设计

v3 的核心定位不变：把 Agent 评测从排名工具升级为诊断工具。v4 在方法论层面引入四项来自最新研究的系统性升级。

3.1 评测环境与任务集构建

整合 InnovatorBench（创新性研究任务）和 AgencyBench（真实长链路工作流）的任务场景，并借鉴 ClawBench 对 OpenClaw 类系统的评测经验，构建覆盖真实场景和创造性问题的统一评测环境。核心设计原则：

  • 工具多样性：集成 API 调用、代码执行、文件操作、浏览器操作等多类工具。
  • Long-Horizon 支持：任务链路涵盖 10–50+ 步决策。
  • 分任务阶段交互：模型需要及时意识到何时需要向人求证。
  • 可控难度：沿数量维度 (N)、复杂度维度 (M) 和交互维度 (K) 进行系统性难度缩放。

v4 NEW 借鉴 TRAJECT-Bench 的任务设计，增加简单/困难配对查询（simple query 直接指定工具，hard query 需间接推理），以及并行/顺序轨迹双结构，量化 Agent 在不同推理路径上的能力差异。

3.2 Agent 能力剖面图（Agent Capability Profile）

v3 中定义了三类属性（任务/行为/效率）和三种分析视角（模型维度/属性维度/桶维度）。v4 在两个方面进行系统性升级：

升级 A：形式化分桶分析框架（来自 InterpretEval）

v3 的分桶分析是描述性的（"在工具链深度 ≥10 时成功率从 94% 降到 61%"）。v4 引入 InterpretEval 的形式化分析框架，将三种分析视角数学化：

Model-wise Analysis：
  Sρ(m, a) = Spearman(performance_m, bucket_a) — 模型 m 在属性 a 上的性能-属性相关性
  Sσ(m, a) = Std(performance_m across buckets of a) — 模型 m 在属性 a 上的性能离散度

Attribute-wise Analysis：
  ζ(a) = mean_m(Sρ(m, a)) — 属性 a 的任务无关偏差（所有模型是否都受此属性影响）
  ρ(a) = std_m(Sρ(m, a)) — 属性 a 的任务依赖性（不同模型受此属性影响程度的差异）

Bucket-wise Analysis：
  自诊断：argMax_b / argMin_b performance(m, a, b) — 模型 m 在属性 a 的最强/最弱桶
  对比诊断：argMax_b |performance(m1, a, b) - performance(m2, a, b)| — 两模型差距最大的桶

统计检验：Friedman's test (跨模型)，Wilcoxon signed-rank (配对模型)，p < 0.05

这一形式化使"Claude 在长链路上比 GPT-5 弱"不再是定性描述，而是可量化、可统计检验的诊断结论。

升级 B：轨迹感知的工具使用评估（来自 TRAJECT-Bench）

v3 的行为属性主要关注工具复用率、错误恢复等宏观行为。v4 新增一组针对工具使用轨迹的细粒度评估维度：

工具选择Trajectory Exact Match (EM)Agent 选择的工具序列与参考轨迹完全匹配的比例最强模型仅 23.5%
工具选择Inclusion Rate参考轨迹中的工具至少被包含在 Agent 输出中的比例达 67.2%，说明"知道用什么但不精确"

3.3 失败传播链分析（Failure Cascade Tracing）

v3 提出了四步方法：轨迹分段 → 逐段评估 → 因果回溯 → 传播模式分类。v4 在此基础上引入 CodeTracer 的层级化方法，实现从"扁平回溯"到"树状诊断"的升级。

升级 C：层级化轨迹树构建（来自 CodeTracer）

CodeTracer 的核心思路是将扁平的 Agent 轨迹重组为三层树结构：

  1. Evolving Extraction（增量抽取）：对每一步 action，识别其 goal（要做什么）、evidence（基于什么信息）、outcome（结果如何），形成结构化的步骤描述。
  2. Tree Indexing（树索引）：将步骤聚合为 Sub-goal 节点，Sub-goal 聚合为 Goal 节点，形成 Goal → Sub-goal → Action 三层树。
  3. Diagnosis（诊断）：在树上标注每个节点的状态（成功/失败/部分成功），自上而下定位失败的最小子树。

v3 vs v4 的关键区别：v3 的因果回溯是沿时间线反向搜索第一个异常步骤；v4 的树状诊断是沿目标层级逐层定位失败子树。后者的优势在于：(1) 不依赖线性时间顺序，可处理并行子目标；(2) 提供更可解释的诊断输出（"数据预处理子目标失败"比"第 12 步出错"信息量更大）；(3) CodeTracer 实证达到 48% step-level F1，提供了可复现的基线。

融合后的失败传播链分析流程：

  1. 轨迹树构建：用 CodeTracer 的 Evolving Extraction + Tree Indexing 方法，将轨迹组织为层级树。
  2. 失败起点定位（Failure Onset Localization）：在树上标注失败起点——最早开始偏离正确路径的 Sub-goal 节点。
  3. 传播路径追踪：从失败起点沿树的兄弟/子节点追踪影响扩散路径。
  4. 传播模式分类：保留 v3 的分类体系（立即崩溃/延迟爆发/渐进退化），增加 CodeTracer 发现的新模式。

3.4 交互判断力评估（Interaction Judgment）

v3 的交互判断力框架（False Confidence / Excessive Inquiry / Effective Inquiry）及其三个核心指标（检查点召回率/交互精确率/交互F1）保持不变。v4 在此基础上增加元认知校准的量化深度：

v4 NEW 引入 TRAJECT-Bench 的 Traj-Satisfy 评分方法，对 Agent 主动交互的质量进行 0-10 LLM 评分，与二值化的 checkpoint recall/precision 形成互补——前者捕捉"问得好不好"，后者捕捉"该不该问"。

同时，CodeTracer 的 Evidence-to-Action Gap 为元认知缺口提供了新的量化手段：Agent 已经收集到了做出正确决策所需的信息，但未能转化为正确行动——这恰好对应 v3 中"知道不确定但不验证"的 RQ7。

3.5 难度分级与双模式评测

v3 的三维难度缩放 (N × M × K) 保持不变。v4 增加来自 TRAJECT-Bench 的缩放瓶颈诊断：

v4 NEW 缩放瓶颈分析：TRAJECT-Bench 发现 Agent 性能在工具链长度从 3 增加到 5 时出现断崖式下降（EM 从 35% 降至 12%），而非线性退化。v4 要求在难度缩放实验中，不仅报告各难度级别的绝对成绩，还要计算相邻级别间的性能衰减曲线，识别每个模型的缩放瓶颈点（performance cliff）。

同时引入 TRAJECT-Bench 的简单/困难配对查询设计：

  • Simple Query：直接指定工具名或明确描述功能需求（"用 X 工具做 Y"）。
  • Hard Query：间接描述需求，需要 Agent 从语义推理到具体工具（"帮我分析这个数据的趋势"→ 需推理使用统计分析工具）。

Simple/Hard 配对的性能差值直接量化了 Agent 的工具发现能力——是否只能在被明确指示时才能正确使用工具。

3.6 自动评估器

v3 的三层架构（基于规则 + LLM-as-Judge + 过程奖励模型）保持不变。v4 引入 AlpacaEval-LC 的去偏方法论，对 LLM-as-Judge 层进行系统性升级：

升级 D：评估去偏（来自 AlpacaEval-LC）

AlpacaEval-LC 的核心发现是：LLM 评估器对输出长度有系统性偏好，导致"更长的回答 = 更高的评分"。Agent 评测中存在类似的混淆变量：

Agent 评测中的混淆变量偏差方向对应 AlpacaEval-LC 概念
Token 消耗量更多 token → LLM judge 可能给更高的"thoroughness"分Length bias
工具调用步数更多步骤 → 看起来"更认真"Length bias (step-level)
Reasoning 内容长度更长的 thinking → 看起来"更深思熟虑"Length bias (reasoning-level)
验证步骤数量更多测试 → 看起来"更谨慎"Verbosity bias

v4 的去偏方法：

Step 1：拟合带混淆变量的 GLM：
  q(y=1 | m, x, confounders) = logistic(θ_m - θ_base + φ·f(confounders) + (ψ_m - ψ_base)·γ_x)
  其中 confounders = {token_count, step_count, reasoning_length, test_count}
  f(·) = tanh(·/std) 归一化

Step 2：计算 Controlled Direct Effect（去偏评分）：
  Score_controlled(m) = logistic(θ_m - θ_base + (ψ_m - ψ_base)·γ_x)
  （移除 φ·f(confounders) 项 = 移除混淆变量的直接效应）

去偏目标：Token-Consumption Controlled Success Rate / Step-Count Controlled Success Rate

这一方法的实际价值在于：当模型 A 用 50K token 得 80 分、模型 B 用 20K token 得 75 分时，去偏后可能发现 B 的真实能力高于 A——因为 A 的 5 分优势来自于 token 消耗的混淆效应而非真实能力差异。

v4 NEW Gameability 评估：参照 AlpacaEval-LC，v4 新增评估指标的可博弈性测试——通过 prompt 操纵让 Agent 改变输出风格（更冗余/更简洁），测量评分变化幅度。理想的评测指标应对此类操纵不敏感。AlpacaEval-LC 将 gameability 从 26% 降至 10%。
