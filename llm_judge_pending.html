§1为什么需要 LLM-Judge

v4 目录里 96 条指标按检测方式分三类：AUTO-RULE（确定性规则）、AUTO-STAT（统计聚合）、AUTO-JUDGE（语义判断）。前两类已经全部跑完，第三类 30 条都没跑。下面这张对比能直接说明区别——

AUTO-RULE   已跑完，不需要 LLM

举例 M07 先读后改率：edit_file 前是否对同一文件 open_file 过？

for step in trajectory:
    if step.action == "edit_file":
        path = step.params["path"]
        prev_opens = {s.params["path"]
            for s in trajectory[:step.idx]
            if s.action == "open_file"}
        if path in prev_opens:
            hit += 1
        total += 1
return hit / total
  • 输入是结构化字段：action_type、params、observation.success
  • 判断规则可以写成代码——是否相等、是否包含、计数、比值
  • 660 条轨迹 30 秒跑完，确定性输出

AUTO-JUDGE   需要语义理解，没跑

举例 #69 推理-行动一致性：reasoning 里说的计划，跟实际调的工具是不是同一回事？

# 规则代码写不出来，因为：
reasoning = """先看一下数据格式，
  如果是 jsonl 的话再决定后面怎么处理"""

action = open_file("data/train.csv")

# 一致吗？文件不是 jsonl 但他确实在"看格式"。
# 必须有人/LLM 读自然语言才能判断。
  • 输入是自然语言文本：reasoning、observation message、人类反馈、judge 评语
  • 判断需要语义对齐、因果推理、意图理解——规则代码无法表达
  • 只能让另一个强 LLM 当 judge，每步打标签
为什么之前的 47 条都能跑完？ 它们要么是规则（开关型 / 计数型 / 比值型，全在结构化字段上），要么是统计（Spearman ρ、Kruskal H、GLM 系数等数值聚合）。轨迹 JSON 里 action_type/params/success/error_message 这些字段是机器写的，规则代码可以稳定提取。但 thinking、content、observation_message 是模型生成的自然语言，里面藏的是意图、假设、不确定性、反思——这些必须语义理解才能量化。
跑这 30 条需要的实际工程量：660 条轨迹 × 平均 ~80 个 react step ≈ 52,800 次 judge 调用（每条指标）。粗算用 4o-mini 每次调用 ¥0.005，每条指标约 ¥260；30 条全跑约 ¥8,000 + 6 小时。其中 9 条还需要先标注因果图 / 参考轨迹 / 子任务依赖图，工程量翻几倍。建议分两批：先跑 6-8 条 judge 提示直接、价值高的，再决定剩下的。
