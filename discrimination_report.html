Agent·Eval 指标区分度分析报告

InnovatorBench + AgencyBench + SWE-bench 三基准联合分析 — 哪些指标真正有用？
日期: 2026-04-25 轨迹总数: 27,485 数据源: 3个 + 3个 + 3个 指标: 34 + 30 + 28

0 这份报告在说什么

我们设计了一套多维度评估体系（Agent·Eval），试图从"过程行为"来评估 AI Agent 的能力，而不仅仅看最终得分。但一个关键问题是：我们设计的这些过程指标，到底有没有用？能不能区分好模型和差模型？

为了回答这个问题，我们在三个 benchmark 上跑了全套指标，并用5种统计方法来检验每个指标的"区分度"：

方法回答的问题怎么读结果
Kruskal-Wallis H + η²这个指标在不同模型之间有没有显著差异？η² > 0.06 = 有差异，> 0.14 = 差异很大
Spearman ρ这个指标跟"做得好不好"（最终得分）有没有关系？|ρ| > 0.3 = 中等相关，> 0.5 = 强相关
PCA 主成分分析这些指标总共在测几个独立维度？PC越分散 = 指标越不冗余（好事）
Random Forest哪些过程指标最能预测最终得分？重要性 > 0.05 = 有实际预测价值
Kendall τ按这个指标给模型排名，跟按得分排名一致吗？|τ| > 0.4 = 排名高度一致

1 三个 Benchmark 对比

先看数据全貌：三个 benchmark 的规模和特征差异很大，这影响了指标的表现。

InnovatorBench

  • 655 条轨迹 / 15 个模型 / 40 个任务
  • 平均 160步 / 轨迹，930万 tokens
  • 平均耗时 4.1小时
  • 得分范围 0-100，平均 11.7
  • 有得分的轨迹仅 44%（289/655）
  • 数据格式：节点树（react/planning/judge）
  • 轨迹特点：长时间、高复杂度的开放任务

AgencyBench

  • 165 条轨迹 / 22 个模型 / 37 个任务
  • 平均 17次工具调用，8.6万字符输出
  • 平均耗时 10分钟（仅19条有时间数据）
  • 得分范围 0-50，平均 15.4
  • 有得分的轨迹 68%（112/165）
  • 数据格式：纯文本agent_output + subtask分数
  • 轨迹特点：结构化子任务、短周期的编程任务

SWE-bench Environment Trajectories (NEW)

  • · 26,665 条去重轨迹 / 3 个数据源 / 来自 38 个JSONL文件
  • · 平均 90次工具调用，16万字符输出
  • · 数据源：difficult_cls（3,090条）、codeact-glm47（11,984条）、code-sweagent-glm47（11,591条）
  • · Pass分类：pass0(59)、pass1(2,298)、count_1(3,199)、count_1_2(7,330)、count_2(1,455)、unknown(12,324)
  • · 无显式得分——按文件名中的pass级别分组
  • · 数据格式：JSONL（messages + tool_calls），工具包括 bash/str_replace_editor/submit/think/task_tracker
  • · 轨迹特点：大规模代码修复，agent在真实代码仓库中进行诊断和修复
关键差异：InnovatorBench 是"马拉松"（每条轨迹跑4小时、160步），AgencyBench 是"短跑"（10分钟、5个子任务），而 SWE-bench 是"中距离竞速"（约90次工具调用，在真实代码仓库中修bug）。SWE-bench 的独特之处在于没有显式评分，分组依据是文件名中编码的pass级别和数据来源（不同的Agent框架：sweagent vs codeact）。这使得SWE数据主要用于分析不同框架间的行为差异，而非预测最终得分。

4 关键问题：过程指标能预测最终得分吗？

这是整个分析最核心的问题。如果过程指标不能预测得分，那我们设计的评估体系就没有实际价值。

99.3%

答案是：能，而且预测力很强。

在 InnovatorBench 上，仅用过程指标就能解释 87.6% 的得分方差。在 AgencyBench 上更是高达 99.3%。这意味着：模型的行为模式几乎完全决定了它的最终表现。

6 反直觉的发现

数据中有几个违背常识预期的现象，值得特别关注。
#发现数据可能的解释
1 花更多时间/token的模型得分更高 M20 ρ=+0.32, M19 ρ=+0.29
(InnovatorBench)		 在长时间复杂任务上，"愿意坚持"是优势。弱模型往往半途而废，强模型持续迭代直到成功。这不意味着效率越低越好——而是说在当前任务难度下，能力不足导致的"快速放弃"才是真正的低效。
2 死锁步骤比例高的模型排名也高 M14 τ=+0.56
(InnovatorBench)		 强模型做更多复杂操作，偶尔会卡住，但最终能解决。弱模型从不死锁——因为它们根本没走到那一步。就像只有会开车的人才会遇到堵车，不会开车的人根本不上路。
3 AgencyBench上得分退化多的模型总分反而更高 M71 ρ=+0.39
(AgencyBench)		 强模型做了5个子任务所以有4次退步机会，弱模型只做了2个子任务所以最多退步1次。退步次数多是因为做得多，不是因为做得差。
4 M03子任务数在两个benchmark上方向相反 InnovatorBench ρ=−0.15
AgencyBench ρ=+0.43		 InnovatorBench中，子任务数是模型自行规划的——弱模型过度拆分反而是能力不足的表现。AgencyBench中，子任务数由题目决定——做了更多子任务的自然得分更高。同一个指标在不同场景下含义完全不同。

8 下一步建议

优先级行动预期收益
P0 补充 JUDGE 类指标：当前15个全是 RULE 型（基于规则自动计算），缺少需要 LLM 判断的软指标（如代码风格、方案创新性、沟通清晰度） 覆盖"软能力"维度，当前指标主要衡量"做了什么"但不衡量"做得有多好"
P0 统一数据格式：AgencyBench 缺少 token 消耗、精确时长等关键字段。建议推动数据采集标准化 让跨benchmark对比更有意义；当前40%的指标因数据缺失无法跨benchmark验证
P1 修复 M10 盲目重试：当前检测逻辑过于严格（要求完全相同的错误字符串），导致全为0。改为语义相似度匹配 盲目重试是重要的失败模式，但当前算法捕获不到
P1 在更多benchmark上验证：当前只在2个benchmark上跑过，样本量（尤其AgencyBench仅165条）还不够大 增强统计置信度，发现更多跨场景稳定的指标
